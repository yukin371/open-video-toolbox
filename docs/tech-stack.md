# 技术选型

## 1. 结论

首选技术栈：

- 平台与语言：`.NET 8 + C#`
- 桌面 UI：`Avalonia`
- 核心调用：`ffmpeg` / `ffprobe`
- 封装能力预留：`mkvmerge`
- 测试：`xUnit`
- 日志：结构化日志 + 原始进程输出双轨保留

## 2. 为什么不是纯逆向驱动

- 旧工具的实现价值低于它的工作流价值
- 逆向得到的是历史行为，不等于今天合理的架构
- 维护成本的关键不在“能不能重现”，而在“能不能持续演进”

因此逆向应只用于补充以下信息：

- 旧预设语义
- 任务编排方式
- 配置文件结构

## 3. 为什么选 .NET + Avalonia

- Windows 桌面工具链成熟，适合这类本地工作台
- 语言与生态对进程调用、文件系统、异步任务支持成熟
- 后续如需跨平台，Avalonia 迁移成本可控
- 对复杂桌面交互、列表、日志窗口、预设编辑器都足够实用
- 同时适合 CLI-first 工作流和外部 AI 代理编排

## 4. 分层建议

- `Core`
  - 不依赖 UI，负责媒体模型、任务模型与应用服务
- `Cli`
  - 作为最早可运行入口，用于验证媒体处理能力并提供结构化输出
- `Desktop`
  - 承载交互与可视化，不承载核心业务规则

## 5. 关键工程约束

- 命令构建必须纯函数化，避免隐式读取 UI 状态
- 外部工具适配统一收口，不能散落调用
- 预设格式版本化，避免后续迁移困难
- 所有关键任务和结构化输出都应能序列化并回放
- 仓库内不内置 AI provider，AI 由外部代理通过 CLI 编排
- 二次手动剪辑优先通过文本计划文件完成，不先上完整时间线编辑器

## 6. 暂不选择的路线

- Electron
  - 迭代快，但该项目更需要强类型、长期维护和本地任务调度能力
- Rust + Tauri
  - 长期潜力高，但当前启动成本高于收益
- 纯 Python
  - 适合脚本，不适合作为长期桌面主干方案
