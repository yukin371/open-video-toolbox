# ADR-0003: Use edit.json For Manual Second-Pass Editing Before Building A Full Editor

日期：2026-04-15

## 状态

Accepted

## 背景

产品希望覆盖视频剪切、音频分离、字幕生成、BGM / 多轨音频、鼓点节奏等高频功能。同时，外部 AI 代理只能负责生成建议，实际剪辑结果往往还需要人工二次修正。

如果直接进入完整时间线编辑器，会带来过早的 UI 复杂度；如果完全不支持人工修正，又会让 AI 生成结果难以落地。

## 决策

- 在完整编辑器出现前，统一使用 `edit.json` 作为剪辑计划交换格式。
- 外部 AI 代理可以生成 `edit.json`。
- 用户可以直接手工修改 `edit.json` 做二次修正。
- CLI 的 `render` 类命令消费 `edit.json` 并执行最终导出。
- GUI 若未来存在，也必须优先复用 `edit.json`，而不是另起一套计划结构。

## 结果

正面影响：

- AI 生成结果可审计、可版本化、可复现
- 人工修正成本低，不必等待完整时间线编辑器
- CLI 与未来轻量 UI 可以共享同一边界

代价：

- 需要谨慎设计 `edit.json` 字段语义
- 复杂编辑能力会被推迟到后续阶段

## 不采纳的方案

- 完全不支持人工二次修正
  - 原因：AI 生成结果在真实创作中经常需要细调
- 直接做完整多轨时间线 GUI
  - 原因：当前阶段 UI 复杂度远高于收益

## 后续动作

- 为 `edit.json` 设计最小 schema
- 让第一批编辑子命令围绕 `edit.json` 生成或消费
- 等 CLI 工作流稳定后，再评估是否需要轻量可视化编辑器
