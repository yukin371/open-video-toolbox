# 开发原则

## 1. 架构优先级

优先级从高到低如下：

1. 可维护
2. 可测试
3. 可观察
4. 可扩展
5. 功能数量

## 2. 设计原则

- 先模型，后界面
- 先边界，后实现
- 先日志，后优化
- 先 CLI 验证，后 GUI 打磨

## 3. 编码原则

- 一个对象只表达一个明确职责
- 命令构建不要直接读写磁盘
- 外部进程调用必须支持取消和超时
- 错误信息要带上下文，不返回“执行失败”这类空话

## 4. 测试原则

- 预设映射用单元测试覆盖
- 命令生成用快照测试或文本断言覆盖
- 任务编排用集成测试覆盖
- GUI 只测关键交互，不把测试预算浪费在样式细节上

## 5. 版本原则

- 预设格式必须带版本号
- 破坏性变更要写迁移说明
- 文档不允许长期落后于代码结构

## 6. 协作原则

- 一次提交只解决一类问题
- 重要决策先写文档再扩散实现
- 对外部依赖增加保持克制
- 遇到“先这样写以后再重构”的情况，要把债务显式记录

## 7. 契约与快照原则

- CLI 结构化输出属于外部契约，不能把改 snapshot 当作默认收尾动作
- 修改 `src/OpenVideoToolbox.Cli.Tests/snapshots/*.json` 时，必须同时说明：
  - 为什么要改
  - 这次变化是否属于 breaking change
  - 是否需要同步 `CHANGELOG.md`
- 若输出包含机器相关路径、临时目录或其他环境噪音，优先做结构归一化断言，而不是扩大 golden file 噪音
- `stdout` 与 `--json-out` 的内容必须保持一致；一旦不一致，应先修行为，再考虑是否更新快照
- 新增 snapshot 覆盖时，优先选择：
  - 机器无关
  - 高频被外部 AI 或脚本消费
  - 在 CI 中可稳定重放
