# E2 生态与可持续演进执行计划

最后更新：2026-04-22

## 背景

- `H1 -> H2+T1 -> T2 -> P1 -> E1` 已完成，CLI 契约、模板插件边界、发布主链和文档分层已具备继续演进基础。
- `D1 Desktop MVP` 目前仍受 `edit.json schema v1` 低频变更窗口约束，最早也要到 2026-05-22 之后才适合重新判断是否启动。
- 因此如果不等待 `D1`，当前最稳妥的下一阶段就是 `E2`：继续补齐生态、兼容性、分发渠道与长期维护护栏。

## 当前定位

本文档是 `E2` 的当前执行计划，用于把长期总纲里的 `E2` 拆成一组可落地工作包。

- 它不替代长期路线文档。
- 当前仓库已进入 `E2` 的连续推进，`E2-A1`、`E2-A2` 已形成实质交付，`E2-A3` 正在补齐贡献路径。
- 后续如当前优先级再次变化，应由 `docs/roadmap.md` 重新声明活跃工作面。

## E2 的首轮目标

首轮 `E2` 不追求“大而全”，只做四类最能提升可持续性的工作：

1. 契约兼容性检查可自动化
2. 分发渠道有清晰优先级与评估结论
3. 社区模板 / 插件贡献路径可独立操作
4. 外部依赖、性能与安全有最小基线

## 本轮不做

- 不启动 Desktop MVP
- 不引入运行时代码插件
- 不直接推进 `edit.json schema v2`
- 不上远程插件市场
- 不做远程渲染 / 任务队列

## 工作包

### E2-A1: 契约兼容性护栏

**目标：** 把“breaking change 检测”从人工约定推进到稳定自动化。

**范围：**

- 明确 contract snapshot golden files 的维护规则
- 明确哪些输出变化必须视为 breaking change
- 评估是否在 CI 中增加“黄金文件变更审查门槛”
- 补一份面向贡献者的契约变更流程说明

**优先 owner：**

- `OpenVideoToolbox.Cli.Tests`
- `docs/README.md`
- `CHANGELOG.md`

**验收标准：**

1. 贡献者能明确知道什么变化需要更新 golden files
2. breaking change 的定义不只存在于 changelog，而能被测试和流程共同约束
3. 至少有一条可复用的 CI / review 规则保护 CLI 契约

**专项清单：**

- `docs/plans/2026-04-22-e2-a1-contract-compatibility-guardrails.md`

### E2-A2: 分发渠道评估与选择

**目标：** 在 winget / Homebrew / NuGet global tool 之间收敛一个首选方向，而不是长期停留在“以后再说”。

**范围：**

- 评估三种渠道对当前发布产物的适配性
- 明确每种渠道的维护成本、平台覆盖、版本同步方式
- 选定首个值得尝试的分发渠道
- 若当前不值得做，明确“不做”的原因与重新评估条件

**优先 owner：**

- `.github/workflows/release.yml`
- `docs/README.md`
- `docs/PROJECT_PROFILE.md`

**验收标准：**

1. 至少产出一份带结论的渠道评估，不再停留在开放问题
2. 文档中明确首选渠道或明确后置理由
3. 若选择实施，实施范围应限定在单一渠道，不并行摊大

**专项清单：**

- `docs/plans/2026-04-22-e2-a2-distribution-channel-evaluation.md`

### E2-A3: 社区模板 / 插件贡献路径

**目标：** 让不了解内部实现的贡献者也能独立提交合规模板插件。

**范围：**

- 补齐模板插件贡献说明
- 约定社区模板示例、审核入口、最小校验清单
- 评估是否需要社区模板目录约定或仓库模板样板
- 明确“什么样的扩展属于模板插件，什么样的不属于”

**优先 owner：**

- `docs/plugin-development-guide.md`
- `examples/plugin-example/`
- `docs/plans/2026-04-19-template-plugin-entry-boundary.md`

**验收标准：**

1. 新贡献者不需要口头解释也能创建和自测模板插件
2. 社区模板提交的最小要求有文档可依
3. 不会因为社区贡献而推动 `Core` 接受私有执行语义

**专项清单：**

- `docs/plans/2026-04-22-e2-a3-community-plugin-contribution-path.md`

### E2-A4: 依赖 / 性能 / 安全最小基线

**目标：** 把长期维护最容易失控的三类问题先建立最小护栏。

**范围：**

- 依赖兼容性基线：ffmpeg / ffprobe / whisper-cli / demucs
- 性能基线：选一组代表性工作流记录执行时间与资源占用的观察方式
- 安全基线：检查外部工具调用中的路径、参数、输出文件覆盖与日志保留规则

**优先 owner：**

- `OpenVideoToolbox.Core/Execution`
- `docs/external-dependencies.md`
- `docs/ARCHITECTURE_GUARDRAILS.md`

**验收标准：**

1. 至少有一份依赖兼容性矩阵或验证约定
2. 至少有一份最小性能观察基线，不要求一开始就做完整 benchmark 系统
3. 至少有一份安全检查清单，覆盖高风险外部调用边界

**专项清单：**

- `docs/plans/2026-04-22-e2-a4-runtime-baseline.md`

## 建议执行顺序

```text
E2-A1 契约兼容性护栏
   │
   ├─ E2-A2 分发渠道评估
   ├─ E2-A3 社区插件贡献路径
   └─ E2-A4 依赖 / 性能 / 安全基线
```

说明：

- `E2-A1` 应最先开始，因为它直接影响后续所有社区贡献与分发的稳定性。
- `E2-A2`、`E2-A3`、`E2-A4` 可以并行，但每次只应真正推进一个主工作包，避免把 `E2` 又做成大杂烩。

## 建议的首个切入点

如果现在正式进入 `E2`，建议先做 **E2-A1 契约兼容性护栏**。

原因：

- 成本最低
- 风险最小
- 最能放大现有 contract snapshot 的价值
- 也最能支撑后续社区模板、包管理器和长期维护

## 完成判定

只有当以下结果全部出现时，才可认为 `E2` 首轮真正产生了阶段价值：

1. 契约变更流程不再只靠口头约束
2. 至少一个生态面问题从“开放问题”变成“有结论”
3. 社区贡献者能在少量维护下独立完成一类扩展
4. 长期维护风险至少在依赖、性能、安全三者之一上有显式基线
